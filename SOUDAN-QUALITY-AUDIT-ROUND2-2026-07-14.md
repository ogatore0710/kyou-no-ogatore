## ① 「頭が締め付けられる」(緊張型頭痛の定番表現)が、胸の赤旗キーワードに巻き込まれて誤爆する

**種別**: redFlags.kwの範囲が広すぎることによる誤爆（前回とは逆方向の問題＝「穴」ではなく「広すぎ」）

**なぜ**: 2026-07-14の前回監査で、胸の締め付け感・冷や汗（心筋梗塞・狭心症のサイン）を拾うために`締め付けられる`・`締め付け感`という**部位を限定しない**赤旗語を追加した（本人YES済み・コミットdbbd61c）。
ところがこの2語は「胸」を含まない文にもヒットしてしまう。そして`zutsuu`（頭痛もち）インテント自身のmitateには、まさに同じ言葉づかいがある:

> zutsuu.mitate: 「**締めつけられるような頭痛**なら、犯人は頭じゃなく首と肩にいることが多いよ。」

つまり、ユーザーが「頭が締め付けられるように痛い」とごく普通の緊張型頭痛の言い方をしただけで、`zutsuu`インテント（首肩ケアの動画・見立て）にたどり着けず、問答無用で「病院へ」という赤旗回答に迂回させられてしまう。これは前回追加した赤旗語の**適用範囲の見落とし**で、実測で確認済み:

```
"胸が締め付けられる感じで冷や汗が出る" => redFlag (意図どおり)
"頭が締め付けられるように痛い"          => redFlag (誤爆・本来はzutsuuインテントに行くべき)
"肩がこって頭が締め付けられる感じがする" => redFlag (誤爆・同上)
```

**対処案（実測で検証済み）**: 部位を限定しない`締め付けられる`・`締め付け感`の2語を削除し、代わりに胸に限定した`胸の締め付け`・`むねのしめつけ`を追加する（既存の`胸が締め付け`・`むねがしめつけ`と合わせて、胸文脈のみカバー）。

```
削除: 締め付けられる / 締め付け感
追加: 胸の締め付け / むねのしめつけ
```

**再実測（削除+追加後）**:
```
"胸が締め付けられる感じで冷や汗が出る"     => redFlag (維持・OK)
"胸の締め付け感がある"                     => redFlag (新規カバー・OK)
"頭が締め付けられるように痛い"             => 通常(zutsuuへ)・OK
"肩がこって頭が締め付けられる感じがする"   => 通常・OK
"締め付けられるような頭痛がする"           => 通常・OK
"ズボンで胃が締め付けられる"               => 通常・OK
```

**判断してほしいこと**: この2語削除＋2語追加（実質は「胸限定」への絞り込み）でよいか（YES/NO）

## ② 「腰が痛くて夜中に目が覚める」のような自然な語順だと、赤旗の夜間痛キーワードから漏れる

**種別**: redFlags.kw追加（語順のゆらぎ対応）

**なぜ**: 既存の赤旗には「夜中に痛」「痛くて目が覚め」「痛みで夜中に目」等があるが、いずれも**特定の語順の連続した一続きの文字列**でないとヒットしない。ユーザーが症状→時間の順で話す「腰が痛くて夜中に目が覚める」は、"痛くて"と"目が覚め"の間に"夜中に"が割り込むため、既存のどの赤旗語とも連続一致せず素通りする。しかも素通りした結果、フォールバックにも行かず**無関係な`nemuri`（寝つきが悪い）インテントに誤着地**してしまう（腰痛の受診サインが完全に無視される、前回穴より一歩踏み込んだ実害）。

```
"腰が痛くて夜中に目が覚める"   => 現状: nemuri(睡眠)インテントに誤着地（赤旗にもyoutsuuにも届かない）
"夜中に腰が痛くて目が覚める"   => 現状: 赤旗ヒット（語順がたまたま合えば拾える）
```

**対処案**: よくある語順ゆらぎを追加でカバーする（`痛くて夜中に目`・`いたくてよなかにめ`等）。ただし**これは対症療法**で、日本語の語順は無限のバリエーションがあるため全てを拾いきることはできない（例えば「夜中に目が覚めるくらい腰が痛い」のように"痛い"がさらに離れると今回の追加でも拾えないことを実測確認済み）。根本対応（「夜中」「目が覚め」「痛」の3語が近接して**どんな順でも**現れたら赤旗、という近接判定ロジック）は、kwリストの追加では実現できずエンジン側のロジック変更が必要になるため、今回はスコープ外として記録のみ。

**判断してほしいこと**:
1. よくある語順ゆらぎの追加語（`痛くて夜中に目`・`いたくてよなかにめ`・`夜中に目が覚めるほど痛`・`よなかにめがさめるほどいた`）を今すぐ足すか（YES/NO・気休め程度の対症療法である点は了承の上で）
2. 根本対応（近接判定ロジックへのエンジン変更）は今回着手しない、でよいか（YES=今回は見送り／NOなら次のエンジン改修セッションで検討）

## ③ 否定形の文で赤旗・crisisが誤って立つ（false positive方向・前回とは逆の見落とし角度）

**種別**: 判断確認のみ（コード変更は非推奨・現状維持を推奨した上で確認を仰ぐ）

**なぜ**: 赤旗・crisisの一部キーワードは「しびれ」「動悸」「妊娠」「腫れ」「吐き気」「骨折」「手術」「めまい」のような**単語1つだけ**で構成されている。これらは文中に否定（「〜はない」「〜していない」）が付いていても、単語自体は含まれているため機械的にヒットしてしまう。実測:

```
redFlag=true | しびれはないです
redFlag=true | めまいは特にありません
redFlag=true | 動悸はないです
redFlag=true | 手術はしていません
redFlag=true | 妊娠はしていません
redFlag=true | 腫れはありません
redFlag=true | 吐き気はないです
redFlag=true | 骨折はしていないと思う
crisis=true  | 死にたいわけじゃないけど疲れた
crisis=true  | 自殺なんて考えたことないです
```

これは日本語の否定を検知する仕組みがない（そもそも本エンジンは文脈理解をしない単純な部分一致）ことによる、構造的な限界。

**なぜ変更を推奨しないか**: この誤爆の方向は「本来は通常回答でよい人に、安全側の受診案内・crisis窓口案内が余分に出る」という**安全側に倒れる誤り**。見落とし方向（本当に危険なのに何も出ない）とは違い、実害は小さい。逆に「否定語があれば赤旗を解除する」ロジックを追加すると、「しびれはないと思ってたけど今日はある」のような複雑な文で**本当に必要な赤旗を消してしまう**リスクの方が大きく、安全設計としては今のままの方が手堅いと判断している。

ただし`crisis`の2例（「死にたいわけじゃない」「自殺なんて考えたことない」）は、明確に希死念慮を否定している人に「いのちの電話」の案内が出るため、UXとして違和感を覚える可能性がある。とはいえ「つらい気持ちを教えてくれてありがとう。ひとりで抱えないでほしいな」という文面自体は否定文脈でも攻撃的・失礼にはならない設計になっており、実害は小さいと考えている。

**判断してほしいこと**: 現状維持（コード変更なし）でよいか（YES=現状維持／NOなら対応方針を指定してほしい。ただし対応する場合も「否定語辞書」を作る作業が発生し、医療安全に関わる判断＝本人確認が必須になる）
